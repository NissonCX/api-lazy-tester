# AI 接口测试工作流

本文档定义 AI Agent 执行 Java 后端接口测试的标准工作流。

## 一、输入约定

AI Agent 执行接口测试时，需要以下输入：

### 1.1 必要输入

| 输入 | 说明 | 来源 |
|------|------|------|
| token | 用户手动提供的认证 Token | 用户直接提供 |
| baseUrl | API 基础 URL | 用户直接提供 |

### 1.2 可选输入

| 输入 | 说明 | 来源 |
|------|------|------|
| 目标接口 | 具体要测试的接口路径 | 用户指定 |
| 模块/Controller | 要测试的 Controller | 用户指定 |
| operationId | OpenAPI 中的操作 ID | 用户指定 |
| OpenAPI 文件 | Swagger/OpenAPI 导出文件 | 项目中的 .yaml/.json 文件 |
| 源码路径 | Controller/DTO 源码目录 | 用户指定 |

### 1.3 输入示例

```
用户：测试用户模块的登录接口
Token: eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.xxx
BaseUrl: http://localhost:8080
```

## 二、信息源优先级

当 AI Agent 需要获取接口定义时，按以下优先级使用信息源：

### 2.1 优先级顺序

1. **OpenAPI / Swagger 导出文件**（最高优先级）
   - 项目中的 openapi.yaml / openapi.json
   - Swagger 导出的 .json 文件
   - 已有的 API 文档

2. **Controller 源码**
   - Controller 类及其注解
   - 方法签名和参数
   - @RequestMapping、@GetMapping、@PostMapping 等

3. **DTO / Entity 源码**
   - 类定义和字段
   - Bean Validation 注解（@NotNull、@Size、@Pattern 等）
   - Swagger 注解（@ApiModel、@ApiModelProperty 等）

4. **枚举和常量**
   - 枚举类定义
   - 常量类
   - JavaDoc 注释

### 2.2 使用原则

- 优先使用 OpenAPI 定义，因为它已经整合了接口和参数信息
- 当 OpenAPI 不完整时，用源码补充
- 不要凭空编造参数名或类型

## 三、测试步骤

### 3.1 定位接口

根据用户输入，定位要测试的接口：

1. 如果用户提供了具体路径，直接定位
2. 如果用户提供了 Controller 名称，查找对应的 Controller
3. 如果用户提供了模块名，查找该模块下的所有接口

### 3.2 读取接口定义

获取接口的完整定义：

- 请求方法（GET/POST/PUT/DELETE）
- 请求路径
- 请求头参数
- 路径参数
- 查询参数
- 请求体结构
- 响应状态码
- 响应体结构

### 3.3 提取参数结构

解析参数定义：

1. **简单类型参数**：直接生成测试值
2. **复杂对象参数**：递归解析对象结构
3. **数组参数**：生成包含 0、1、多个元素的数组
4. **枚举参数**：从枚举类中获取可选值

### 3.4 生成最小合法请求

根据接口定义，生成最小合法请求：

```json
{
  "username": "test",
  "password": "123456"
}
```

原则：
- 使用最短的有效字符串
- 使用最小的有效数字
- 使用最简单的有效对象
- 可选参数不填

### 3.5 生成负例

根据参数约束，生成负例测试：

| 约束类型 | 负例 |
|----------|------|
| @NotNull | 传 null 或不传 |
| @NotEmpty | 传空字符串或空数组 |
| @Size(min=2) | 传单字符 |
| @Pattern(regexp) | 传不符合格式的字符串 |
| @Min(0) | 传负数 |
| @Max(100) | 传超过 100 的数字 |

### 3.6 执行测试

执行请求并记录结果：

1. 设置请求头（Content-Type、Authorization 等）
2. 发送请求
3. 记录响应状态码
4. 记录响应体
5. 记录响应时间

### 3.7 输出报告

生成结构化测试报告：

```json
{
  "接口": "POST /api/user/login",
  "测试用例": [
    {
      "名称": "正常登录",
      "请求": {...},
      "响应状态": 200,
      "响应体": {...},
      "结果": "通过"
    },
    {
      "名称": "用户名为空",
      "请求": {"username": "", "password": "123456"},
      "响应状态": 400,
      "结果": "通过"
    }
  ],
  "总结": "通过 2/2"
}
```

## 四、默认风险边界

AI Agent 在执行测试时，默认遵守以下风险边界：

### 4.1 低风险操作（默认允许）

- 读取 OpenAPI 定义文件
- 读取源码文件
- 发送 GET 请求
- 发送只读 POST 请求（查询类）
- 生成测试报告

### 4.2 中风险操作（需要确认）

- 发送需要认证的请求（使用用户提供的 token）
- POST 创建资源（需确认是否会影响数据）
- PUT 更新资源（需确认是否会影响数据）

### 4.3 高风险操作（默认禁止）

- DELETE 删除资源
- 批量操作
- 导入/导出大量数据
- 修改配置
- 执行可能影响生产环境的操作

## 五、高风险操作确认规则

当需要执行高风险操作时，AI Agent 必须：

1. **明确告知用户**：说明将要执行的操作和风险
2. **等待确认**：请求用户明确确认后再执行
3. **提供回滚方案**：如果可能，说明如何回滚

确认示例：

```
⚠️ 即将执行删除操作：DELETE /api/user/123
这将永久删除用户数据，是否确认继续？（yes/no）
```

## 六、输出格式规范

### 6.1 测试报告格式

推荐使用 JSON 格式，便于后续处理：

```json
{
  "测试时间": "2026-03-27T10:00:00Z",
  "基础URL": "http://localhost:8080",
  "接口总数": 5,
  "测试结果": [
    {
      "接口路径": "/api/user/login",
      "方法": "POST",
      "状态": "通过",
      "用例数": 3,
      "通过数": 3
    }
  ],
  "详情": [...]
}
```

### 6.2 单接口测试格式

```json
{
  "接口": "POST /api/user/login",
  "用例": [
    {
      "名称": "最小合法请求",
      "请求体": {"username": "a", "password": "123"},
      "预期状态": 200,
      "实际状态": 200,
      "结果": "通过"
    },
    {
      "名称": "缺少必填参数",
      "请求体": {"password": "123"},
      "预期状态": 400,
      "实际状态": 400,
      "结果": "通过"
    }
  ]
}
```

## 七、工作流状态机

```
[开始]
   ↓
[获取输入] → 缺少必要输入？ → [请求补充]
   ↓
[定位接口]
   ↓
[读取定义]
   ↓
[生成用例]
   ↓
[执行测试]
   ↓
[生成报告]
   ↓
[结束]
```

## 八、扩展点

以下内容可根据具体实现进行扩展：

1. **测试框架集成**：可以集成 JUnit、TestNG 等
2. **报告格式**：可以支持 HTML、Markdown 等格式
3. **断言库**：可以使用 JSONAssert、Hamcrest 等
4. **并发测试**：可以支持压力测试
5. **录制回放**：可以录制请求并回放

## 九、注意事项

1. **不要假设**：所有参数信息都应从定义文件中获取
2. **不要猜测**：不确定的参数类型要询问用户
3. **记录来源**：测试报告中要说明信息来源（OpenAPI/源码）
4. **保留证据**：保存请求和响应，以便排查问题
5. **尊重约束**：严格按照 Bean Validation 注解生成负例
